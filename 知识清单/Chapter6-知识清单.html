6.1 网络应用体系结构

客户机/服务器结构（C/S） 必考

服务器（server）：7×24 小时提供服务，拥有永久性可访问的地址/域名，利用大量服务器实现可扩展性。客户机（client）：与服务器通信以使用其提供的服务，间歇性接入网络，可能使用动态 IP 地址，客户机之间不会直接通信。典型例子：Web（浏览器 ↔ Web 服务器）。

纯 P2P 结构 必考

纯 P2P 结构没有永远在线的服务器，任意端系统（节点，peer）之间可以直接通信；节点间歇性接入网络，且可能改变 IP 地址。优点：高度可伸缩；缺点：难于管理。

混合结构（Hybrid） 重点

混合结构试图同时利用 C/S 与 P2P 两者的优点。典型例子——Napster：文件传输采用 P2P 结构（节点间直接传输文件），文件搜索采用集中式 C/S 结构（每个节点向中央服务器登记自己的内容并提交查询请求）。

6.2 网络应用通信原理

进程间通信与 Socket 必考

应用是运行在不同端系统上的进程之间通过交换消息实现的：客户机进程——发起通信的进程；服务器进程——等待通信请求的进程。同一主机上的进程通信由操作系统提供的进程间通信机制解决；不同主机上的进程通信依赖消息交换。

Socket 是进程发送/接收消息所经过的接口，类似寄信：发送方将消息交给（门外的）传输基础设施，由其负责送达接收方主机的门外，接收方再从门外取走消息。Socket 是应用层与传输层之间的 API——应用开发者控制 Socket 的应用层一侧，操作系统控制传输层一侧（传输协议的选择、参数设置）。

进程寻址：IP 地址 + 端口号 必考

不同主机上的进程通信需要为每个进程分配标识符。仅靠 IP 地址不足以定位进程，因为同一主机上可能同时有多个进程需要通信；因此还需要端口号（port number）为主机上每个需要通信的进程分配标识，例如 HTTP 服务器默认使用端口 80，邮件服务器（SMTP）默认使用端口 25。进程的完整标识符为IP 地址 + 端口号。

应用层协议的三要素 重点

网络应用需遵循应用层协议，分为公开协议（由 RFC 定义，允许互操作，如 HTTP、SMTP）与私有协议（多数 P2P 文件共享应用采用）。协议规定三方面内容：消息类型（type）（请求消息/响应消息）、消息的语法/格式（syntax）（消息中有哪些字段、每个字段如何描述）与字段的语义（semantics）及规则（rules）（进程何时/如何发送与响应消息）。

网络应用对传输服务的需求 必考

需求维度说明典型应用
数据丢失/可靠性某些应用能容忍一定丢失，某些要求 100% 可靠网络电话（容忍丢失）；文件传输、Telnet（要求可靠）
时间/延迟只有延迟足够低才"有效"网络电话、网络游戏
带宽部分应用需要最低带宽保证；弹性应用可适应任何带宽网络视频（需要带宽）；email（弹性）

TCP 与 UDP 提供的传输服务对比 必考

能力TCPUDP
连接方式面向连接：需先建立连接无连接
可靠性可靠传输不可靠传输
流量控制有（防止发送方发送过快压垮接收方）无
拥塞控制有（网络负载重时限制发送速率）无
时延/带宽保障不提供不提供
易错：TCP 和 UDP 均不提供时延保障与最小带宽保障，二者的差异主要体现在连接建立、可靠性、流量控制与拥塞控制上。

6.3 域名解析系统（DNS）

DNS 的作用与为什么不用集中式设计 必考

Internet 上主机/路由器的标识存在两种方式：IP 地址（用于寻址）与域名（如 www.hit.edu.cn，便于人记忆）。DNS 是完成域名到 IP 地址映射的应用层协议，本质是由多层命名服务器构成的分布式数据库。除主机名到 IP 地址的翻译外，DNS 还提供主机别名、邮件服务器别名、负载均衡（为同一 Web 服务器的多个 IP 地址进行分发）等服务。

不采用集中式 DNS 的原因：单点失效问题、流量问题（网络规模扩大导致的性能瓶颈）、距离问题（集中式服务器可能离用户很远，产生显著传播时延）、维护性问题（集中式数据库难以及时更新维护）。

域名层次结构与分布式层次式数据库 必考

域名按层次划分：根（Root） → 顶级域名（TLD）（如 com、org、net、edu、gov、cn）→ 二级域名 → 三级域名 ……不同的域可以由不同管理机构划分与维护。对应地，DNS 服务器分为三类：

服务器类型职责
根域名服务器全球共 13 个（编号 a~m），若不知道映射则指向相应的顶级域名服务器
顶级域名服务器（TLD）负责 com、org、net、edu 等通用顶级域名和国家顶级域名（cn、uk、fr 等）
权威域名服务器组织自身的域名服务器，提供组织内部主机的解析服务，由组织或服务提供商维护

此外还有本地域名解析服务器——它不严格属于层级体系，每个 ISP 至少拥有一个本地域名服务器（默认域名服务器），主机发起的 DNS 查询首先发送到本地域名服务器，再由其作为代理（proxy）转发给层级式域名服务器系统。

DNS 查询过程：递归查询与迭代查询 必考

DNS 查询包含两种方式：递归查询——主机向本地域名服务器发起查询后，后续所有查询均由本地域名服务器代理完成，最终直接返回结果给主机（主机与本地域名服务器之间只需一次请求/应答）；迭代查询——本地域名服务器依次向根、顶级域名、权威域名服务器发起查询，每次都得到"下一步该问谁"的答复，再自行发起下一次查询，最终获得结果。典型查询路径：本地域名服务器 → 根域名服务器 → 顶级域名服务器 → 权威域名服务器。

易错：区分"用户主机 → 本地域名服务器"这一段与"本地域名服务器 → 上级服务器"这一段各自采用的是递归还是迭代，二者可以采用不同的方式组合（常见组合为：客户端到本地服务器递归，本地服务器到上级服务器迭代）。

DNS 记录缓存与更新 重点

只要域名服务器获得了某个域名-IP 映射，即会将其缓存；一段时间后缓存条目失效并被删除。本地域名服务器一般会缓存顶级域名服务器的映射，因此根域名服务器不经常被直接访问。记录的更新依赖通知机制（RFC 2136，Dynamic Updates in the DNS）。

DNS 资源记录（RR） 必考

资源记录格式为 (name, value, type, ttl)。

TypeNameValue
A主机域名IP 地址
NS域（如 edu.cn）该域权威域名服务器的主机域名
CNAME某一真实域名的别名真实域名
MX邮件服务器别名邮件服务器域名

DNS 协议与消息格式 常考

DNS 只有查询（query）和回复（reply）两类消息，且共用相同的消息格式。消息头部字段包括：Identification（16 位查询编号，回复使用相同编号以便匹配）与 flags（标识该消息是查询还是回复、是否期望递归、递归是否可用、是否为权威回答等）。消息正文包含：查询的 name/type 字段、对查询的应答 RR、权威域名服务器记录、可能有用的附加信息。

域名注册流程 了解

以在域名管理机构（如 Network Solutions）注册 networkutopia.com 为例：需向域名管理机构提供权威域名解析服务器的名字和 IP 地址；域名管理机构随后向 com 顶级域名服务器插入两条记录：NS 记录（networkutopia.com, dns1.networkutopia.com, NS）与A 记录（dns1.networkutopia.com, 212.212.212.1, A）；此后在自己的权威域名服务器中，还需为 www.networkutopia.com 添加 A 记录，为邮箱地址添加 MX 记录和 A 记录。

DNS 例题：递归解析的消息数 必考

【例】若本地域名服务器无缓存，采用递归方法解析另一网络某主机域名时，用户主机与本地域名服务器各自发送的域名请求消息数分别为？

【解析】递归查询中，用户主机只需向本地域名服务器发送一次查询，之后由本地域名服务器代理后续所有查询并返回最终结果，因此本地域名服务器对上级发送的查询次数与主机侧无关（可能是多次，但都由本地域名服务器完成，不会让主机重复发起）。就"用户主机、本地域名服务器各自发出的请求消息数"而言，正确答案为：一条、一条——因为题目考察的是主机侧只发一次请求，本地域名服务器作为代理只需向权威链路发出一次递归请求即可获知最终结果（由被查询服务器代理后续查询）。

6.4 FTP 应用

FTP 的控制连接与数据连接分离 必考

FTP（RFC 959）采用 C/S 模型：客户端发起文件传输（上传或下载），服务器为远程主机。FTP 使用两条独立的 TCP 连接：

连接端口用途
控制连接（control connection）21传送控制信息（登录认证、切换目录、发出传输命令），贯穿整个会话，属于"带外（out of band）"控制
数据连接（data connection）20每次文件传输单独建立一条，传输完一个文件后即关闭

FTP 服务器需要维护状态（state）：当前目录、此前的身份认证信息等，这与 HTTP 的无状态设计不同。

易错：不要把 FTP 的"控制连接常驻、数据连接按需建立/关闭"记反；每传输一个文件，服务器就会开启一条新的数据连接，传输完毕后关闭该数据连接，但控制连接保持不变。

FTP 常用命令与响应码 常考

命令以 ASCII 文本形式通过控制通道发送（与 HTTP 类似）。常用命令：USER（声明用户名）、PASS（声明密码）、LIST（返回当前目录文件列表）、RETR filename（下载文件）、STOR filename（上传文件）。响应由状态码和描述短语构成，例如 331（Username OK, password required）、125（数据连接已打开，开始传输）、425（无法打开数据连接）、452（写入文件出错）。

6.5 Email 应用

Email 系统的组成 重点

Email 应用由三部分组成：用户代理（User Agent）——用于读写邮件消息的客户端（如 Outlook、Foxmail、Thunderbird，或 Web 客户端）；邮件服务器（Mail Server）——包含存储用户邮件的邮箱（mailbox）与存放待发邮件的外出邮件队列（message queue）；SMTP 协议——邮件服务器之间传递消息所使用的协议，客户端为发送消息的服务器，服务器端为接收消息的服务器。

SMTP 协议 必考

SMTP（RFC 2821）使用 TCP（端口 25）实现邮件消息的可靠传输，是推式（push）协议。传输过程分三阶段：握手 → 消息传输 → 关闭，采用命令/响应交互模式（命令为 ASCII 文本，响应为状态代码+语句）。SMTP 消息内容只能包含 7 位 ASCII 码。

典型交互：HELO（握手）→ MAIL FROM → RCPT TO → DATA（以单独一行的 "." 结束数据）→ QUIT。

Email 消息格式与 MIME 重点

SMTP 负责消息的传输/交换，而RFC 822 定义了文本消息格式标准：由头部行（header）（To、From、Subject 等）、空白行、消息体（body）组成，消息体只能是 ASCII 字符。为了支持多媒体内容，MIME（RFC 2045、2056）在邮件头部增加额外的行来声明内容类型，包括 MIME-Version、Content-Transfer-Encoding（数据编码方法，如 base64）、Content-Type（多媒体数据的类型/子类型/参数）。

邮件访问协议：POP3 与 IMAP 必考

用户代理需要通过邮件访问协议从邮件服务器获取邮件：

协议特点
POP3（RFC 1939）认证/授权 + 下载，"下载并删除"或"下载并保持"两种模式；无状态（stateless）——若采用"下载并删除"模式，用户更换客户端软件后无法重读该邮件
IMAP（RFC 1730）功能更丰富也更复杂：所有消息统一保存在服务器上，允许用户利用文件夹组织消息，支持跨会话（session）的用户状态（如文件夹名字、文件夹与消息 ID 之间的映射）
HTTP（Web Mail）通过浏览器直接访问邮箱
易错：POP3 是无状态协议，不能跨会话保留文件夹结构等状态信息；能够跨会话保持这些状态的是 IMAP，二者不要混淆。

6.6 Web 应用

URL 与 HTTP 协议概述 必考

网页（Web Page）由多个对象（objects）组成（HTML 文件、JPEG 图片、视频文件、动态脚本等），基本 HTML 文件包含对其他对象的引用链接。对象通过 URL（Uniform Resource Locator，RFC 1738）寻址，格式为 scheme://host:port/path。HTTP（超文本传输协议）遵循 C/S 结构：客户端（Browser）负责请求、接收、展示 Web 对象；服务器（Web Server）负责响应请求、发送对象。HTTP 使用 TCP 作为传输服务，服务器默认在端口 80等待连接请求。

HTTP 无状态特性 必考

HTTP 是无状态（stateless）协议：服务器不维护任何有关客户端过去所发请求的信息。有状态协议实现更复杂：需要维护状态（历史信息），且如果客户或服务器失效，会产生状态不一致，解决这种不一致代价较高。

HTTP 请求/响应消息格式 必考

请求消息由请求行（request line，包含方法如 GET/POST/HEAD、URL、版本）、若干首部行（header lines）（如 Host、User-agent、Connection、Accept-language）、一个额外的回车换行（表示消息结束）组成，整体为 ASCII 文本，人可直接阅读。响应消息由状态行（status line，含协议版本、状态码、状态短语，如 HTTP/1.1 200 OK）、首部行（Connection、Date、Server、Last-Modified、Content-Length、Content-Type 等）、数据部分（请求的 HTML 文件等）组成。

HTTP 方法与状态码 常考

HTTP/1.0 方法：GET、POST。HTTP/1.1 新增：HEAD（请求服务器不要将对象放入响应消息中）、PUT（将消息体中的文件上传到 URL 字段指定的路径）、DELETE（删除 URL 字段指定的文件）。表单上传输入既可用 GET 方法（输入信息附加在 URL 字段上），也可用 POST 方法（输入信息通过消息体上传）。常见状态码：200 OK、301 Moved Permanently、400 Bad Request、404 Not Found、505 HTTP Version Not Supported。

非持久连接与持久连接：响应时间分析 必考

非持久连接（Nonpersistent HTTP，HTTP/1.0 默认）：每个 TCP 连接只传输一个对象，每请求一个新对象都要重新建立连接。响应时间（Response time）定义为：客户端发送小数据包到服务器再返回所经历的RTT，加上文件的发送时间。非持久连接下每个对象的响应时间 = 2×RTT + 响应消息发送时间（1 个 RTT 用于建立 TCP 连接，1 个 RTT 用于发送 HTTP 请求并收到响应消息前几个字节，再加上传输时间）。

持久连接（Persistent HTTP，HTTP/1.1 默认）：服务器发送响应后保持 TCP 连接打开，同一连接上可传输后续的多个 HTTP 消息（多个对象）。

$$\text{响应时间（非持久连接）} = 2 \times RTT + \text{文件传输时间}$$

易错：不要把"非持久连接每对象需要 2 个 RTT"误记为"1 个 RTT"；持久连接下首个对象仍需 1 个 RTT 建立连接，但后续对象在理想流水线情况下每个只需约 1 个 RTT（或接近 0，如果流水线充分重叠）。

并行连接与流水线 重点

并行连接：浏览器可打开多个并行 TCP 连接以同时获取网页的多个对象，缓解非持久连接"每对象 2 RTT"的问题，但会给服务器带来更大开销。非流水的持久连接：客户端必须收到前一个响应后才能发送下一个请求，每个对象仍耗时约 1 个 RTT。流水的持久连接（pipelining，HTTP/1.1 默认）：客户端只要遇到一个引用对象就尽快发出请求，理想情况下收到全部引用对象总共只需约 1 个 RTT。

HTTP/2 常考

HTTP/1.1 存在队首阻塞（Head-of-Line, HOL blocking）问题：服务器按先来先服务（FCFS）顺序响应，若一个大对象排在小对象之前，后面的小对象需要等待大对象传输完成；一旦发生丢包，还会暂停整个对象的传输。HTTP/2（RFC 7540, 2015）为减少多对象请求的响应时延，将每个对象分割为一系列"帧"，不同对象的帧可以交替传输；对象传输顺序基于客户端定义的优先级（而非 FCFS）；支持服务器推送（push）客户端未主动请求的对象。方法、状态码、多数首部行与 HTTP/1.1 保持不变。

HTTP/3 与 QUIC 了解

HTTP/2 仍使用单一 TCP 连接，一旦发生丢包恢复仍会暂停该连接上所有对象的传输（TCP 层面的 HOL blocking 无法根除）。HTTP/3 基于 QUIC（Quick UDP Internet Connections）——一种建立在 UDP 之上的应用层协议：在应用层自行实现可靠性、拥塞控制、连接建立、身份验证与加密等功能；差错控制和拥塞控制采用类似 TCP 的算法；连接建立（可靠性、拥塞控制、身份验证、加密）可在 1 个 RTT 内完成（相比 TCP+TLS 需要 1~2 轮握手，QUIC 将其合并）；多个应用级"流（stream）"复用一个 QUIC 连接，可靠数据传输与安全彼此分离，实现流级别的并行性，从根本上消除 HOL blocking（一个流丢包不会阻塞其他流）。

Web 缓存/代理服务器 必考

目标：在不访问源服务器的前提下满足客户端的 HTTP 请求。作用：缩短响应时间、减少机构/组织的出口流量、在大范围内（Internet）实现有效的内容分发。工作方式：用户设置浏览器通过代理访问 Web；若请求对象在缓存中，直接返回；否则代理向原始服务器发起请求，获取对象后既返回给客户端也在本地保存一份。缓存同时充当客户端（面向原始服务器）和服务器（面向浏览器），一般由 ISP 架设。

Web 缓存示例：命中率对平均时延的影响 必考

【场景】平均对象大小 = 1 Mbit，机构网络中浏览器平均每秒发出 10 个到原始服务器的请求，机构路由器到原始服务器的往返时延 RTT = 2 秒。

方案 1（不缓存，接入带宽 10 Mbps）：接入链路利用率 = 100%（10 Mbps × 1s 恰好被 10 个 1Mbit 对象占满），根据排队论，链路利用率接近 100% 时排队时延趋于无穷大，总时延 ≈ 互联网时延 + 访问时延（可达几分钟） + 局域网时延，用户体验很差。

方案 2（提升接入带宽至 100 Mbps）：接入链路利用率降至 10%，访问时延降至几微秒，总时延 ≈ 2 秒 + 可忽略访问时延，但成本太高。

方案 3（部署本地 Web 缓存，命中率 = 0.4，带宽仍为 10 Mbps）：40% 的请求由缓存立即满足，60% 的请求通过原始服务器满足，接入链路利用率下降到 60%，访问时延可忽略（如 10 微秒）；总的平均时延 ≈ 0.6×2.0 秒 + 0.4×(约 0 微秒) ≈ 1.2 秒，在不增加接入带宽成本的前提下显著降低了平均时延。

易错：接入链路利用率越接近 100%，排队时延不是线性增长而是趋于无穷大（M/M/1 排队模型的特性），这正是方案 1 时延"几分钟"级别的根本原因；不要误以为时延只是简单地按比例增长。

条件性 GET 方法与 Cookie 必考

条件性 GET（Conditional GET）：目标是"如果缓存已有最新版本，则不需要重新发送对象内容"。客户端（缓存）在 HTTP 请求消息中携带 If-modified-since: <date> 声明所持有版本的日期；若服务器判断该版本仍是最新的，则返回 304 Not Modified（响应消息中不包含对象数据），否则正常返回 200 OK 及最新数据。

Cookie（RFC 6265）用于让服务器辨别用户身份、进行会话（session）跟踪，由四部分组成：HTTP 响应消息中的 Set-cookie 首部行、HTTP 请求消息中的 Cookie 首部行、保存在客户端主机上由浏览器管理的 cookie 文件、Web 服务器端的后台数据库。原理：服务器首次响应时用 Set-cookie 分配一个唯一 ID 并在后台创建对应条目；此后客户端每次请求都在 Cookie 首部行带上该 ID，服务器据此识别用户并执行 cookie-specific 的动作。用途：身份认证、购物车、推荐系统；同时也带来隐私问题。

6.7 P2P 应用

纯 P2P 架构特点 重点

纯 P2P（Peer-to-Peer）架构没有服务器，任意端系统之间可以直接通信，节点阶段性接入 Internet 且可能更换 IP 地址。

文件分发：C/S 与 P2P 的时间对比 必考

设：服务器上传能力 $u_s$，节点 $i$ 上传能力 $u_i$，节点 $i$ 下载能力 $d_i$，文件大小 $F$，节点数 $N$，$d_{min}$ 为下行带宽最小节点的下载能力。

C/S 方式：服务器需串行发送 $N$ 个副本，耗时 $NF/u_s$；下行带宽最小的客户下载耗时 $F/d_{min}$。

$$D_{c\text{-}s} \geq \max\{NF/u_s,\ F/d_{min}\}$$

该式表明 $D_{c\text{-}s}$ 随 $N$ 线性增长。

P2P 方式：服务器只需上传 1 个副本，耗时 $F/u_s$；下行带宽最小节点下载耗时 $F/d_{min}$；服务器与所有节点共同上传 $NF$ 比特数据总耗时 $NF/(u_s + \sum u_i)$。

$$D_{P2P} \geq \max\{F/u_s,\ F/d_{min},\ NF/(u_s+\textstyle\sum u_i)\}$$

虽然分母中的总量项 $NF/(u_s+\sum u_i)$ 分子也随 $N$ 线性增长，但每个新加入的节点也会带来自己的上传能力，使总服务能力同步增长，因而 P2P 方式的分发时间增长速度远低于 C/S 方式，体现了 P2P 架构的可扩展性优势。

易错：不要认为 P2P 分发时间与 $N$ 无关；它同样含有随 $N$ 增长的项，只是因为节点同时贡献上传带宽，增长速度显著慢于 C/S 方式，而非完全恒定。

BitTorrent 协议 必考

文件被分割为大小为 256KB 的数据块（chunk）。tracker：跟踪参与某个 torrent（交换同一文件数据块的节点集合）的所有节点；新节点加入 torrent 时先无 chunk，从 tracker 获取节点列表并与部分节点（邻居）建立连接，逐渐积累 chunk，同时也需要向其他节点上传 chunk；节点获得完整文件后可以选择离开或留下继续做种。

获取 chunk 的策略：稀缺优先（rarest first）——节点定期查询各邻居持有的 chunk 列表，优先请求当前最稀缺的 chunk。发送 chunk 的策略：互惠 / 一报还一报（tit-for-tat）——节点向当前发送速率最快的 4 个邻居（top 4）发送 chunk，每 10 秒重新评估一次 top 4；同时每 30 秒随机选择一个其他节点发送 chunk，称为"乐观疏通（optimistically unchoke）"，使新的潜在优质伙伴有机会被发现。

互惠机制的正反馈：(1) A 对 B 进行乐观疏通，向其发送数据；(2) A 因此成为 B 的 top-four 数据提供方之一；(3) B 也将 A 纳入其 top-four。上传速率越高，越能找到更好的交易伙伴，从而越快获取完整文件——这正是 tit-for-tat 的激励意义。

P2P 搜索信息：集中式索引、查询洪泛与层次覆盖网络 常考

方案代表特点
集中式索引Napster节点加入时向中央目录服务器登记 IP 地址与内容；查询集中在服务器完成，文件传输是 P2P 的；存在单点失效、性能瓶颈、版权问题
查询洪泛（Query flooding）Gnutella完全分布式的覆盖网络（overlay network）：节点只对自己共享的文件建立索引；查询消息在覆盖网络中转发（洪泛），命中后沿反向路径返回结果；节点一般邻居数少于 10 个

分布式哈希表（DHT） 必考

DHT（Distributed Hash Table）是一个分布式的 P2P 数据库，存储 (key, value) 对。核心问题是如何将 key 分配给对等方（peer）：将每个 key 转换为整数（对原始键做散列（hash）），为每个对等方分配一个整数标识符 ID（取值范围 $[0, 2^n-1]$），再依据某种"距离"规则将 (key, value) 存放到最近的 peer 上。本课程约定："最近"是指 key 的直接后继者（successor）。

【示例】设 $n=4$，节点集合为 {1,3,4,5,8,10,12,14}：key=13 的直接后继是节点 14；key=15（超出最大节点 14）的直接后继"环绕"回节点 1。

圆形 DHT（Circular DHT）：每个对等方只知道自己的直接后继与前驱，构成"覆盖网络"；当存在 $N$ 个节点时，朴素查询报文的平均复杂度为 $O(N)$（逐跳转发直到找到负责该 key 的节点）。带捷径的圆形 DHT：每个对等方额外维护若干"捷径"邻居的 IP 地址，可将查询消息数从线性降低到 $O(\log N)$。

节点流失（Peer churn）处理：由于节点会随时加入或离开（churn），每个对等体需要知道其两个（而非一个）后继的地址，并定期 ping 检查它们的活动状态；若直接后继离开，则用间接后继顶替为新的直接后继，并向其询问新的第二后继（例如 Peer5 突然离开：检测到离开后使 Peer4 的直接后继变为 Peer8，并询问 Peer8 的直接后继，将其设为 Peer4 的新第二后继）。

6.8 Socket 编程

Socket API 概览 了解

应用编程接口（API，Application Programming Interface）是应用进程控制权与操作系统控制权相互转换的系统调用接口。历史上常见的三种网络 API：BSD UNIX 定义的 Socket 接口、微软 Windows 上的 Winsock、AT&T UNIX V 定义的 TLI（Transport Layer Interface）。Socket 面向 C/S 通信模型，是事实上的工业标准，绝大多数操作系统都支持。Socket 对外以 IP 地址 + 端口号标识通信端点，对内由操作系统用一个小整数（套接字描述符，socket descriptor）管理。

核心 Socket API 函数与调用流程 了解

常用函数：socket()（创建套接字）、bind()（绑定本地 IP 地址与端口号，客户端通常无需调用）、listen()（服务器端置 TCP 套接字为监听状态并设置连接请求队列大小）、connect()（客户端与目标建立 TCP 连接，或为 UDP 客户端指定服务器端点地址）、accept()（服务器从连接请求队列中取出一个请求并创建新套接字与之通信，仅用于 TCP）、send()/recv()（用于 TCP 套接字或已 connect 的 UDP 套接字）、sendto()/recvfrom()（用于未 connect 的 UDP 套接字）、closesocket()（关闭套接字）。TCP 典型调用顺序：服务器 socket → bind → listen → accept（阻塞）→ recv/send → closesocket；客户端 socket → connect → send/recv → closesocket。

注：该部分为课程简要介绍内容（PPT 标注"编程（略）"），一般不作为笔试考查重点，仅需理解 Socket 是应用层与传输层之间的接口，以及各函数的基本职责。

易错点汇总

  • DNS：区分递归查询与迭代查询发生的位置（主机-本地服务器段 vs 本地服务器-上级服务器段）。
  • UDP 分用只依据目的 IP+端口二元组；TCP 分用依据源/目的 IP+端口四元组。
  • POP3 无状态、不能跨会话保留文件夹结构；IMAP 才支持跨会话状态维护。
  • HTTP/1.0 默认非持久连接，HTTP/1.1 默认持久连接（且默认使用流水线）。
  • Web 缓存示例中，接入链路利用率趋近 100% 时排队时延趋于无穷大，不是线性增长。
  • P2P 文件分发时间同样含有随 $N$ 增长的项，只是增长速度慢于 C/S，并非与 $N$ 无关。
  • DHT 中"最近"约定为 key 的直接后继者，注意环绕（wrap-around）情形的计算。

本章总结

应用层是网络体系结构的最高层，直接面向用户需求，其协议设计需要综合考虑体系结构选择（C/S、P2P、混合）、进程寻址与复用、传输层服务需求等因素。DNS 作为支撑性的名字解析服务贯穿几乎所有网络应用；FTP、Email（SMTP/POP3/IMAP）、Web（HTTP）是三类经典的 C/S 应用，各自围绕"如何高效、可靠地组织消息交换"设计协议细节；P2P 应用（BitTorrent、DHT 等）则展示了如何利用节点自身资源实现可扩展的分布式系统。
