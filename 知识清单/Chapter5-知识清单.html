5.1 传输层服务

传输层的核心作用 必考

传输层协议为运行在不同主机上的进程之间提供逻辑通信机制。发送方将应用层递交的消息分成一个或多个报文段（Segment）并向下交给网络层；接收方将收到的 Segment 组装还原为消息，并向上交给应用层。传输层协议只在端系统中运行，网络核心中的路由器不处理传输层信息。

传输层 vs. 网络层 重点

网络层提供主机之间的逻辑通信机制；传输层提供应用进程之间的逻辑通信机制，位于网络层之上，依赖网络层服务，并对网络层服务进行（可能的）增强。

家庭类比（补充解释）：应用进程如同房子里的孩子，应用消息如同信封里的信，主机如同房子，传输层协议如同孩子之间的约定，网络层协议如同邮政服务——邮政服务只负责把信从一栋房子送到另一栋房子，具体信要转交给哪个孩子由屋内的约定（传输层）决定。

Internet 传输层协议 必考

协议可靠性特性
TCP可靠、按序交付拥塞控制、流量控制、连接建立
UDP不可靠交付基于"尽力而为（Best-effort）"的网络层，不做可靠性方面的扩展

两种协议均不保证时延和带宽。

5.2 传输层多路复用/分解

为什么需要复用与分解 必考

如果某一层的一个协议实体要直接为上层的多个协议实体提供服务，就需要复用（multiplexing）与分用（demultiplexing）机制：发送端复用——从多个 Socket 收集数据，为每块数据封装上头部信息生成 Segment 后交给网络层；接收端分用——依据头部信息将收到的 Segment 交给正确的 Socket（即正确的进程）。

无连接分用（UDP） 必考

UDP 的 Socket 由二元组标识：（目的 IP 地址，目的端口号）。主机收到 UDP 段后，只检查段中的目的端口号，即可将其导向绑定在该端口号上的 Socket；来自不同源 IP 地址和/或源端口号的 UDP 数据包，只要目的二元组相同，都会被导向同一个 Socket。

易错：UDP 分用只看目的端口（及目的 IP），源地址、源端口不参与 Socket 的区分，因此不同来源的数据可能进入同一个 UDP Socket。

面向连接的分用（TCP） 必考

TCP 的 Socket 由四元组标识：（源 IP 地址，源端口号，目的 IP 地址，目的端口号）。接收端利用这全部四个值将 Segment 导向正确的 Socket；服务器可同时支持多个 TCP 连接，每个连接对应各自独立的 Socket（例如 Web 服务器为每个客户端连接创建不同的 Socket）。

易错：TCP 分用的四元组任一项不同都会被视为不同连接，因此同一目的端口（如 80）上可以同时存在多条来自不同客户端（或同一客户端不同源端口）的独立连接。

5.3 UDP 协议

UDP 特点与适用场景 必考

UDP（RFC 768）是无连接的：发送方与接收方之间不需要握手，每个 UDP 段的处理独立于其他段。特点：简单的错误校验；无需建立连接，可减少延迟；实现简单，无需维护连接状态；头部开销小；没有拥塞控制，应用可以更好地控制发送时间和速率。常用于流媒体应用（容忍丢失、速率敏感）、DNS、SNMP。若应用需要可靠性，只能在应用层自行实现（如停等协议、滑动窗口协议）。

易错："UDP 不可靠"不代表 UDP 之上的应用一定不可靠，可靠性可以由应用层协议在 UDP 之上自行叠加实现。

UDP 段格式 重点

UDP 首部固定 8 字节，包含：源端口号、目的端口号、长度（UDP 段总长度，含首部）、校验和，其后是应用数据。

UDP 校验和计算 必考

目的：检测 UDP 段在传输中是否发生错误（如位翻转）。计算范围包括三部分：伪首部（源 IP 地址、目的 IP 地址、全 0 字节、协议号 17、UDP 总长度）、UDP 首部、应用数据。

计算步骤：1）将参与计算的内容视为若干个 16 位整数；2）对这些 16 位整数求和（sum），产生的进位加回到和的末尾；3）将 sum 按位求反（取反码），得到校验和 checksum 并放入校验和字段。接收方按同样方法对收到的段重新计算 16 位整数序列之和；若结果各位全为 1（即 sum=1111111111111111），则判定无错，否则判定有错。

易错：反码求和时产生的最高位进位必须回卷加到和的末尾（回卷相加），遗漏这一步会导致校验和计算错误。

5.4 TCP 协议

5.4.1 TCP 段结构与连接特点 必考

TCP（RFC 793、1122、1323、2018、2581）是点对点（一个发送方、一个接收方）、全双工（同一连接可双向传输数据流）、面向连接（通信前须先建立连接）、提供可靠、按序字节流服务的协议，并具有流水线机制与拥塞控制/流量控制机制。连接状态只在连接两端维护，沿途路由器不维护 TCP 连接状态。TCP 连接包括：两台主机上的发送/接收缓存、连接状态变量、Socket 等。

TCP 段固定首部 20 字节，主要字段：源/目的端口号（各 16 位）、序号（32 位）、确认序号（32 位）、首部长度（4 位，以 4 字节为计算单位）、保留（6 位，值为 0）、6 个标志位（URG/ACK/PSH/RST/SYN/FIN）、接收窗口（16 位，用于流量控制）、校验和（16 位）、紧急指针（16 位，仅 URG=1 时有效）、选项（长度可变，如 MSS、时间戳、SACK）及填充（0～3 字节，补齐 4 字节整数倍）。

标志位含义
URG为 1 时紧急指针字段有效
ACK为 1 时确认序号字段有效
PSH为 1 时要求尽快将数据交付给接收应用进程
RST为 1 时重新建立 TCP 连接
SYN为 1 时表示这是建立新连接的请求段
FIN为 1 时表示请求释放该 TCP 连接

序号与确认序号 必考

序号是该段所携带数据第一个字节在整个字节流中的编号，而不是"段的连续编号"（不是第 1 段、第 2 段这样计数）。建立连接时，双方各自随机选择初始序号。确认序号是接收方期望接收到的下一个字节的序号，采用累计确认方式——该序号之前的所有字节都已被正确接收。TCP 规范并未规定接收方应如何处理乱序到达的段，由具体实现自行决定。

易错：序号编号的对象是"字节"，不是"段"；很多同学误以为第 2 个发送的段序号就该是"2"，实际上要看该段第一个字节在字节流中的绝对位置。

5.4.2 TCP 可靠数据传输 必考

TCP 在 IP 层不可靠服务的基础上实现可靠数据传输，采用流水线机制、累积确认，并使用单一重传定时器。触发重传的事件：超时或收到 3 个冗余（重复）ACK。

RTT 估计与超时间隔设置 必考

SampleRTT：测量从段发出到收到其 ACK 的时间（忽略重传段的测量，因为无法区分该 ACK 是确认原始段还是重传段）。由于 SampleRTT 本身波动较大，TCP 使用指数加权移动平均（EWMA）得到更平滑的估计值：

$$EstimatedRTT = (1-\alpha)\times EstimatedRTT + \alpha \times SampleRTT$$

$\alpha$ 典型值为 0.125。同时还需要估计 RTT 的波动幅度：

$$DevRTT = (1-\beta)\times DevRTT + \beta \times |SampleRTT - EstimatedRTT|$$

$\beta$ 典型值为 0.25。最终超时间隔设置为在 EstimatedRTT 基础上加上一个正比于波动幅度的"安全边界"：

$$TimeoutInterval = EstimatedRTT + 4 \times DevRTT$$

易错：若超时时间设置得过短，会导致不必要的重传（网络本身并未丢包）；若设置得过长，则对段丢失的反应过慢，两者都需要通过 EstimatedRTT 与 DevRTT 动态平衡。

快速重传（Fast Retransmit） 必考

若 TCP 实现仅依赖超时检测丢包，则每次超时都会使超时时间间隔加倍，重发丢失分组之前可能要等待很长时间。发送方通常会背靠背发送多个分组，一旦某个分组丢失，后续分组的到达会引发接收方针对同一（已确认过的）数据发送多个重复 ACK。TCP 据此实现快速重传：若发送方收到对同一数据的 3 个冗余 ACK，则假定该数据之后的段已经丢失，在定时器超时之前就立即重传该段，从而缩短检测丢包、进行重传所需的等待时间。

5.4.3 TCP 流量控制 必考

流量控制是一种速度匹配机制：接收方为该 TCP 连接分配接收缓存（buffer），发送方不应该发送得太多、太快，以至于淹没接收方缓存（缓存溢出）——这可能是因为上层应用从 buffer 中读取数据的速度较慢。

接收方通过在 Segment 首部的 接收窗口（RcvWindow） 字段将自己 buffer 中的可用空间告知发送方：

$$RcvWindow = RcvBuffer - [LastByteRcvd - LastByteRead]$$

发送方需保证自己已发送但还未被确认的数据量不超过接收方通告的 RcvWindow。

易错：当 RcvWindow=0 时，发送方仍需定期发送只携带 1 字节数据的探测报文段，以便及时获知接收方 buffer 是否重新腾出空间，否则双方可能陷入死锁。（补充解释）

5.4.4 TCP 连接管理 必考

TCP 发送方和接收方在传输数据前需要先建立连接、初始化 TCP 变量，即三次握手（Three-way handshake）：

1）Step 1：客户端向服务器发送一个 TCP SYN 段，指定初始序号，不携带数据（客户端进入 SYN_SENT 状态，服务器初始处于 LISTEN 状态）；

2）Step 2：服务器收到 SYN 后，为该连接分配缓存，回复一个 SYNACK 段（同时包含自己选择的初始序号，并对客户端的 SYN 进行确认），服务器进入 SYN_RCVD 状态；

3）Step 3：客户端收到 SYNACK 后回复一个 ACK 段（该段可以携带数据），双方均进入 ESTABLISHED 状态，连接建立完成。

关闭连接采用四次挥手：Step 1，client 发送 TCP FIN 控制段（client 进入 FIN_WAIT_1）；Step 2，server 收到 FIN 后回复 ACK（server 进入 CLOSE_WAIT，client 收到 ACK 后进入 FIN_WAIT_2），此时 server 仍可继续发送数据；Step 3，server 数据发送完毕后发送自己的 FIN（server 进入 LAST_ACK）；Step 4，client 收到 server 的 FIN 后回复 ACK，并进入 TIME_WAIT 状态等待 2 个最大段生存时间（2MSL）后才真正关闭（若在此期间又收到 server 重发的 FIN，会重新发送 ACK）；server 收到最终 ACK 后连接关闭。

易错：四次挥手中 client 收到 server 的 ACK 后连接尚未完全关闭，而是进入 FIN_WAIT_2 等待 server 一侧数据发送完毕后的 FIN；TIME_WAIT 的存在正是为了保证最后一个 ACK 即使丢失、server 重传 FIN 时 client 仍能正确响应，同时让本连接使用过的序号"冷却"，避免与后续新建连接的分组混淆。

5.4.5 TCP 拥塞控制 必考

拥塞控制的问题在于：网络（IP 层）本身不会显式通知发送方发生了拥塞，TCP 发送方需要自行限制发送速率，通过丢包事件（超时或收到 3 个冗余 ACK）来感知网络拥塞，并用拥塞窗口 CongWin 限制已发送但未被确认的数据量：$$LastByteSent - LastByteAcked \le CongWin$$ 发送速率大致为 $CongWin/RTT$ 字节/秒。发生 loss 事件后，发送方降低发送速率。

AIMD：加性增、乘性减 必考

原理：逐渐增加发送速率，谨慎探测可用带宽，直到发生丢包。加性增（Additive Increase）：每个 RTT 将 CongWin 增大 1 个 MSS（属于拥塞避免阶段）；乘性减（Multiplicative Decrease）：发生丢包后将 CongWin 减半。这种"探测--回退"的过程使 CongWin 随时间呈现锯齿状变化。

慢启动（Slow Start） 必考

TCP 连接刚建立时，CongWin 初始值为 1 个 MSS，可用带宽可能远高于初始速率，因此希望快速增长：每收到一个 ACK，CongWin 就增加 1 个 MSS，效果是每个 RTT 使 CongWin 翻倍（指数性增长），直到发生丢包事件或 CongWin 超过 threshold（门限值）为止。

Threshold 与慢启动/拥塞避免的切换 必考

变量 Threshold 决定何时从指数增长（慢启动）切换为线性增长（拥塞避免）：当 CongWin 达到 Threshold 时，切换到拥塞避免阶段。每次 Loss 事件发生时，Threshold 被设为该事件发生前 CongWin 值的一半。

Loss 事件的处理：三次重复 ACK 与超时 必考

两类丢包检测事件的处理策略不同：

1）3 个重复 ACK：说明网络还能够传输一些分组（收到了后续段的 ACK），因此 Threshold=CongWin/2，CongWin=Threshold，直接进入拥塞避免阶段线性增长（即"快速恢复"，TCP Reno 版本行为）；

2）Timeout（超时）：表明拥塞更为严重，因此 Threshold=CongWin/2，CongWin 直接降为 1 个 MSS，重新进入慢启动阶段指数增长（TCP Tahoe 版本对两种情形均如此处理，不做区分）。

易错：3 个重复 ACK 与超时的处理力度不同——前者（TCP Reno）只是"减半后进入拥塞避免"，后者无论 Reno 还是 Tahoe 都要"降到 1 MSS 后重新慢启动"，超时的处理比冗余 ACK 更"严厉"。

TCP 拥塞控制状态总结 高频

状态事件发送方动作
慢启动收到之前未确认数据的 ACKCongWin=CongWin+MSS；若 CongWin>Threshold，转入拥塞避免
拥塞避免收到之前未确认数据的 ACKCongWin=CongWin+MSS×(MSS/CongWin)（每 RTT 增加 1 个 MSS）
慢启动或拥塞避免3 个冗余 ACKThreshold=CongWin/2，CongWin=Threshold，转入拥塞避免（快速恢复）
慢启动或拥塞避免超时Threshold=CongWin/2，CongWin=1 MSS，转入慢启动

例题：拥塞窗口追踪 高频

一个 TCP 连接总是以 1KB 的最大段长（MSS）发送段，发送方有足够多的数据要发送。当拥塞窗口为 16KB 时发生了超时，如果接下来的 4 个 RTT 内的 TCP 段传输都成功，求第 4 个 RTT 内发送的段都被确认后拥塞窗口的大小。

解：超时发生 --> Threshold = 16/2 = 8 KB，CongWin 降为 1 KB，重新进入慢启动。第 1 个 RTT 后：CongWin=2 KB；第 2 个 RTT 后：CongWin=4 KB；第 3 个 RTT 后：CongWin=8 KB（达到 Threshold，慢启动结束，转入拥塞避免）；第 4 个 RTT 后（拥塞避免阶段每 RTT 只加 1 MSS）：CongWin=9 KB。

TCP 吞吐率 重点

忽略慢启动阶段，假定发生 Loss 时 CongWin 大小为 $W$，吞吐率为 $W/RTT$；Loss 后 CongWin 减半为 $W/2$，吞吐率为 $W/2RTT$；因此 TCP 平均吞吐率约为：$$0.75\,W/RTT$$

吞吐率与丢包率 $L$（loss rate）的关系：设 CongWin 从 $W/2$ 增至 $W$ 时发生第一次丢包，则发送的分组数约为 $3W^2/8$，当 $W$ 很大时可推得 $L \approx 8/(3W^2)$，从而：$$Throughput \approx \frac{1.22 \times MSS}{RTT\sqrt{L}}$$

易错：该公式表明在高丢包率场景下要维持高吞吐率，需要极小的丢包率 $L$（例如追求 10Gbps 吞吐率时 $L$ 需低至 $2\times10^{-10}$ 量级），这也是经典 TCP（AIMD）在高速网络场景下需要改进的动因。

TCP 拥塞控制的改进 了解

TCP CUBIC：Linux 默认拥塞控制算法。核心思想是丢包后不像 AIMD 那样线性爬升，而是先快速逼近上次丢包时的窗口值 $W_{max}$，越接近 $W_{max}$ 增长越谨慎（增长量是与当前时间到某一时刻 $K$ 之间距离的立方函数），从而在维持稳定性的同时提高高速网络下的吞吐率。

基于时延的拥塞控制（Delay-based）：不等到发生丢包才降速，而是持续监测 RTT，目标是"让端到端管道恰好装满但不溢出"：若测量吞吐率接近以最小 RTT（RTTmin）计算的无拥塞吞吐率，则线性增大 cwnd；若测量吞吐率远低于无拥塞吞吐率（说明路径已拥塞、排队时延增大），则线性减小 cwnd。

显式拥塞通知（ECN）：一种网络辅助的拥塞控制方式，路由器在 IP 首部 ToS 字段的两个比特中标记拥塞，目的端在 TCP 段的 ECE 位中将拥塞指示带回给发送方，从而无需真的丢包也能让发送方感知拥塞。

TCP 的公平性 重点

若 $K$ 个 TCP 连接共享同一段带宽为 $R$ 的瓶颈链路，理论上每个连接的平均速率应为 $R/K$，且可以证明 AIMD 机制会使多个 TCP 连接的窗口大小逐渐收敛到近似相等（公平）。

公平性面临的挑战：1）UDP 应用通常不使用 TCP，以恒定速率发送且能容忍丢失，不受拥塞控制机制的速率限制，会挤占 TCP 连接的带宽份额，产生不公平；2）并行 TCP 连接（如某些 Web 浏览器为同一页面同时打开多个 TCP 连接）会获得比单连接应用更多的总带宽份额，同样破坏公平性——例如链路速率为 $R$、已有 9 个连接的情况下，新应用若只建立 1 个 TCP 连接将获得约 $R/10$ 的速率，而若建立 11 个并行连接则可以获得约 $R/2$ 的速率。

易错："TCP 公平性"是指同类型、同数量连接下 AIMD 会收敛到平均分配带宽，但这一结论的前提是"单连接对单连接"公平竞争；一旦引入不参与拥塞控制的 UDP 流量或人为增加并行连接数，实际获得的带宽份额就会显著偏离 R/K 的公平分配。

公式速查

RTT 估计（EWMA）：$$EstimatedRTT=(1-\alpha)EstimatedRTT+\alpha\cdot SampleRTT \ (\alpha=0.125)$$

RTT 波动估计：$$DevRTT=(1-\beta)DevRTT+\beta\cdot|SampleRTT-EstimatedRTT| \ (\beta=0.25)$$

超时间隔：$$TimeoutInterval=EstimatedRTT+4\times DevRTT$$

接收窗口（流量控制）：$$RcvWindow=RcvBuffer-[LastByteRcvd-LastByteRead]$$

平均吞吐率：$$Throughput\approx 0.75\,W/RTT \approx \frac{1.22\times MSS}{RTT\sqrt{L}}$$

K 个连接公平共享带宽 $R$：$$\text{每连接平均速率} = R/K$$

易错点汇总

  • TCP 序号编号的对象是字节，不是段本身的序数。
  • UDP 分用用二元组（目的 IP、目的端口），TCP 分用用四元组（源 IP、源端口、目的 IP、目的端口）。
  • UDP 校验和计算的反码求和，最高位进位必须回卷加到和的末尾。
  • 3 个重复 ACK 触发的处理（减半后进入拥塞避免）比超时触发的处理（降到 1 MSS 重新慢启动）更"温和"。
  • 四次挥手中 client 收到 server 的 ACK 后并未立即关闭，还需等待 server 的 FIN；TIME_WAIT 用于应对 ACK 丢失和序号冷却。
  • TCP 公平性 R/K 的结论只在同类型 TCP 连接、无 UDP 流量抢占、无并行连接的前提下成立。

考试重点提示

本章高频考点集中在：TCP/UDP 分用的元组区别、UDP 校验和计算、TCP 段结构字段含义（尤其是序号/确认号/标志位/接收窗口）、RTT 与超时间隔的 EWMA 公式计算、TCP 三次握手与四次挥手的状态迁移、拥塞控制中慢启动/拥塞避免的切换条件与 Threshold 更新规则、拥塞窗口随 RTT 演化的追踪计算、TCP 吞吐率与丢包率关系公式。综合题常将 RTT/超时计算 与 拥塞窗口演化追踪 结合考查，需要按事件顺序（超时/冗余ACK）逐 RTT 推演，不能跳步骤。

本章总结

传输层在网络层"尽力而为"的基础上，为应用进程之间提供更丰富的通信服务：多路复用/分解解决了"一台主机上多个进程共享网络层服务"的问题；UDP 提供简单、低开销、无拥塞控制的不可靠传输，适合时延敏感型应用；TCP 则通过序号/确认号、流水线与累积确认、超时与快速重传机制实现可靠数据传输，通过 RcvWindow 实现流量控制，通过三次握手/四次挥手管理连接生命周期，并通过 AIMD、慢启动、拥塞避免、快速恢复等机制协同实现拥塞控制与连接间的公平性。理解各机制"为什么这样设计"（如累积确认、指数回退、Threshold 减半）比单纯记忆流程更有助于应对综合题。
